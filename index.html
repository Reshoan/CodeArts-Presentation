Step 3 · Develop Code

Branch → commit → merge request

The golden rule: never code on master. Each feature lives on its own branch and merges only after review.

01
Create branch

Chris creates Feature-Store from master, links the work item.

›
02
Commit code

Edits store-network.html + index.html. Commit message references task ID.

›
03
Open MR

Source Feature-Store → Target master. Maggie set as approver.

›
04
Review & merge

Maggie comments, approves the gate, merges. Task auto-marks Resolved.

Why it matters: "function branches + merge requests" keeps master stable and traceable, while linking commits back to work items gives you full audit history.

CodeArts Best Practices
Step 4 · Check Code

Static analysis with CodeArts Check

Before code merges, scan for quality, style, and security issues — with fix suggestions inline.

Four preset tasks

  • phoenix-codecheck-worker — Worker function
  • phoenix-codecheck-result — Result function
  • phoenix-codecheck-vote — Vote function
  • phoenix-sample-javas — JS across the repo

How to extend

  • Open the task → Settings → Rule Sets
  • Default language is Java; enable Python for Worker
  • Click Execute Check & review the dashboard
  • Click Help on any issue for fix suggestions

Standard practice: coding standards + continuous static scanning + bug-fixing inside each sprint = far less remediation cost downstream.

CodeArts Best Practices
Step 5 · Build & Push

Build images, push to SWR

Package every microservice + environment into an image. Same image runs in dev, test, QA, and prod — no environment drift.

Five preset build tasks

  • phoenix-sample-ci — basic build
  • phoenix-sample-ci-test — test images
  • phoenix-sample-ci-worker / -result / -vote

Each build task does

  • Build Vote / Result / Worker images
  • Generate & build Postgres + Redis images
  • Push all 5 to SWR under your phoenix org
  • Replace versions in compose & k8s files, archive to release repo

Auto-trigger setup

  • Run upon code commit — every push to master rebuilds
  • Scheduled execution — pick days & time (e.g. 12:00 daily)

Result: 5 images in SWR (vote, result, worker, postgres, redis) + .yaml + tar.gz files in the release repo, ready to deploy.

Step 6b · Deploy to ECS

Host deploy: phoenix-sample-standalone

An alternative when you don't want a Kubernetes cluster — deploys to a single host via Docker Compose.

Step 1 — Add the host

  • Settings › Basic Resources › Create Host Cluster (phoenix-hostgroup, Linux, Direct)
  • Import the phoenix-ecs instance
  • Auth: root / password / SSH 22
  • Connectivity check turns green

Step 2 — Pre-deploy dependencies

  • Use phoenix-sample-predeploy to install Docker + Docker-Compose
  • Create phoenix-env environment, import host
  • Logs should show both versions printing

Step 3 — Deploy: in phoenix-sample-standalone, set source = build task phoenix-sample-ci, env = phoenix-env, fill in SWR creds, then Save & Deploy. Verify on http://EIP:5000 & :5001.

Step 9 · Cleanup

Release resources you don't need

Pay-per-use means costs continue until resources are released.

ECS

Delete the ECS

Once verification is complete, delete the phoenix-ecs instance and its EIP.

CCE

Delete the CCE cluster

Delete the pay-per-use phoenix-cce cluster and its node pool.

⚠ Warning: released resources cannot be recovered. Make sure you've captured everything worth keeping (artifacts, screenshots, logs) before pulling the plug.

Section 4 · Permissions · Principles

How we'll roll permissions out

Three principles that make role design sane and reusable.

1

Start from system roles

Project Manager, Developer, Tester come pre-built. Don't reinvent — extend.

2

Use "Copy Role"

For team variants (e.g. Big Data Developer = Developer), copy and adjust deltas.

3

Save as templates

Once a role set is good, save it as a permission template and reuse on the next project.

Edge case: Test managers and Testers don't get Repo edit access — but they do need view access for white-box testing. The defaults handle this; don't strip Repo entirely.

Wrap-up

Key takeaways for our team

What we should walk out of this room agreeing on.

★
Adopt the HE2E loop end-to-end

Plan → iterate → continuously deliver. Don't pick and choose stages.

★
Branch + MR is mandatory

Master is sacred. Every change goes through a reviewed MR.

★
Static check inside the pipeline

With pass conditions — bad code shouldn't reach prod.

★
Containerise everything

One image, every environment. Eliminates "works on my machine".

★
Auto-trigger on commit

Build & pipeline kick off on push to master — no manual runs.

★
Security on by default

CTS · whitelists · protected branches · private params from day one.

★
Role templates for every team

Save once, reuse forever — saves weeks per new project.

★
Release pay-per-use cleanly

Always tear down what you no longer need.
